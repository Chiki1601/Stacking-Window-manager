Stacking Window Manager

(Proposed by Marian Veteanu)

This coding challenge will take you back to the early days of microcomputers and put you in the race to build the next user interface for a micro. In practical terms you’re asked to implement a mini stacking window manager using JavaScript and HTML5 CANVAS.

The design is all yours! Please feel free to be original … or just leverage your knowledge from other operating systems GUI programming to implement this.

Important: Try to time-box this project. Unless you plan to build the next Windows / Mac UI, try not to spend more than 1 – 2 evenings on this coding challenge. Focus only on the features that you believe are important to you…

When you send the solution, please describe:

  • Features of your Window Manager
  • The main idea behind your design
  • What is the developer experience for the user of your GUI system / library
About my implementation...

My implementation is quite basic, a far cry from a real-life, Windows-like front-end.
But in any case, below I’ve enumerated the main features:

Stacking

  • Windows are stacked on top of each other
  • You can make any window top-most, by clicking on it
  • The active window stands out visually, by using a different color scheme
  • You can close any window

Dragging

  • You can move any window by dragging its title bar
  • Over the title bar, the mouse cursor becomes a "move" shape

Resizing

  • You can resize any window, by dragging borders and corners
  • All four borders, and all four corners can be dragged
  • Over borders and corners, the mouse cursor becomes a "resize" shape

Painting

  • Windows are responsible for painting their frame (title bar, close button, and borders)
  • They receive a translated and clipped paint context:
    • Their upper-left corner is seen at coordinates (0, 0) -- no matter their actual position on the canvas
    • They cannot paint outside their area, even if they try to
  • To paint the client area, they receive an optional, developer-provided paint function (see below)

Developer experience

  • Developer is responsible for painting the client area of the windows
  • They can provide a callback function, which is called by the windows manager when painting is due
  • The callback receives a paint context which is translated and clipped:
    • The client area’s upper-left corner is seen at coordinates (0, 0) -- no matter the window’s actual position on screen
    • The callback cannot paint outside of the client area, even if it tried to

Callback paint function

  • As an example, I’ve provided a paint function which does the following:
    • Creates a random number of balls
    • Bounces the balls around, using random speeds for horizontal/vertical movement
    • The balls have random radiuses, colors, and are created at random coordinates inside the client area
